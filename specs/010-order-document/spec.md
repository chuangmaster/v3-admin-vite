# Feature Specification: 商品訂購單文件產生

**Feature Branch**: `010-order-document`  
**Created**: 2026-02-18  
**Status**: Draft  
**Input**: User description: "商品訂購單文件產生 - 商品訂購單提供客戶確認訂單，該文件由前端產生，位置可以相同於列印按鈕的選項，可以先瀏覽再進行列印，效果同出貨單等，由商品明細來組裝訂購人聯絡資訊、訂購明細和商品預購定金須知等內容"

## Clarifications

### Session 2026-02-18

- Q: 訂購單預覽應該如何呈現給使用者？ → A: 在彈出視窗（modal/dialog）中顯示預覽，與現有出貨單 (ShippingLabelPreview.vue) 的實作方式一致，使用 ElDialog 元件
- Q: 訂購單中的付款紀錄應該如何呈現？ → A: 顯示付款記錄詳細列表（包含每筆付款的日期、金額、付款方式），類似出貨單
- Q: 訂購單中的商品明細應該包含哪些欄位？ → A: 根據訂單類型動態顯示：(1) 預購訂單：僅顯示品牌、款式；(2) 現貨訂單：顯示品牌、款式、配件
- Q: 商品明細是否需要包含商品名稱、數量、單價欄位？ → A: 是，預購訂單顯示：品牌、商品名稱、款式、數量、單價；現貨訂單顯示：品牌、商品名稱、款式、配件、數量、單價
- Q: 商品明細過長時，列印分頁應該如何處理？ → A: 自動分頁，使用 CSS break-inside 確保每個商品項目完整不跨頁

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 產生並預覽訂購單 (Priority: P1)

業務人員在確認客戶訂單後，需要產生一份訂購單文件供客戶檢閱。業務人員從商品訂單管理頁面點擊「產生訂購單」按鈕（位於列印選項附近），系統會自動從商品明細中組裝資料，產生包含訂購人資訊、訂購明細和定金須知的完整文件，並在瀏覽器中預覽。

**Why this priority**: 這是核心功能，讓業務人員能夠產生標準化的訂購單文件，提供給客戶確認訂單內容，是整個流程的必要起點。

**Independent Test**: 可以透過選擇一筆商品訂單，點擊產生訂購單按鈕，檢查是否能正確顯示包含所有必要資訊的預覽文件，即可完整測試此功能。

**Acceptance Scenarios**:

1. **Given** 業務人員在商品訂單管理頁面查看某筆訂單，**When** 點擊「產生訂購單」按鈕，**Then** 系統以彈出視窗方式顯示訂購單預覽，包含訂購人姓名、電話、訂購明細、付款紀錄列表和定金須知
2. **Given** 業務人員查看包含多筆付款紀錄的訂單，**When** 產生訂購單，**Then** 付款紀錄區塊顯示每筆付款的詳細資訊（日期、金額、付款方式）
3. **Given** 業務人員查看預購訂單，**When** 產生訂購單，**Then** 商品明細顯示品牌、商品名稱、款式、數量、單價，不顯示配件欄位
4. **Given** 業務人員查看現貨訂單，**When** 產生訂購單，**Then** 商品明細額外顯示配件欄位
5. **Given** 業務人員查看訂購人未提供 Line ID 或銀行帳號的訂單，**When** 產生訂購單，**Then** 對應欄位保持空白，不顯示預設值或錯誤
6. **Given** 訂購單預覽已開啟，**When** 業務人員檢視文件排版，**Then** 排版格式與出貨單（ShippingLabelPreview）一致

---

### User Story 2 - 列印訂購單 (Priority: P2)

業務人員在預覽訂購單確認無誤後，需要將訂購單列印出來或儲存為 PDF，提供給客戶簽署或存檔。業務人員在預覽畫面中點擊列印按鈕，觸發瀏覽器的列印對話框，可以選擇實體列印或儲存為 PDF。

**Why this priority**: 這是文件產出的最終步驟，讓訂購單能夠以實體或電子形式交付給客戶，是完成業務流程的關鍵動作。

**Independent Test**: 可以透過產生訂購單預覽後，點擊列印按鈕，檢查是否能正確觸發列印對話框且版面適合 A4 紙張，即可完整測試此功能。

**Acceptance Scenarios**:

1. **Given** 業務人員正在預覽訂購單，**When** 點擊列印按鈕，**Then** 瀏覽器顯示列印對話框，預覽版面適合 A4 紙張
2. **Given** 業務人員在列印對話框中，**When** 選擇列印或儲存為 PDF，**Then** 列印輸出包含完整的訂購單內容，不遺漏任何欄位
3. **Given** 業務人員列印訂購單，**When** 檢視列印輸出，**Then** 品牌標的、表格框線、文字對齊等視覺元素清晰可見

---

### Edge Cases

- 當訂購人的 Line ID 或銀行帳號末五碼為空時，對應欄位保持空白
- 當訂單包含多筆付款紀錄時，顯示完整的付款記錄列表（每筆付款的日期、金額、付款方式）
- 當訂單沒有付款紀錄時，付款紀錄區塊仍顯示但內容為空
- 當訂購單內容過長（例如商品明細超過 20 項）時，使用 CSS 自動分頁，確保每個商品項目不跨頁
- 當使用者在不同裝置（桌機、平板）上產生訂購單時，預覽視窗自適應裝置寬度，列印輸出統一為 A4 格式
- 預購訂單的商品明細不顯示配件欄位，現貨訂單顯示配件欄位

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系統必須提供「產生訂購單」功能入口，位置與其他列印按鈕（如出貨單）相同
- **FR-002**: 系統必須從商品訂單的商品明細中取得資料，用於組裝訂購單內容
- **FR-003**: 系統必須在訂購單中顯示訂購人聯絡資訊，包含訂購人姓名、電話、Line ID（可選）、銀行帳號末五碼（可選）
- **FR-004**: 系統必須在訂購單中顯示訂購明細，根據訂單類型動態顯示不同欄位：
  - 預購訂單：品牌、商品名稱、款式、數量、單價
  - 現貨訂單：品牌、商品名稱、款式、配件、數量、單價
- **FR-005**: 系統必須從付款紀錄計算並顯示付款紀錄詳細列表（包含每筆付款的日期、金額、付款方式）
- **FR-006**: 系統必須在訂購單中顯示完整的「商品預購定金須知」條款，包含以下內容：
  - 確認訂購後收取 50% 訂購金額為定金的說明
  - 定金退還條件說明
  - 手工製作商品的瑕疵容忍說明
  - 商品鑑定與保固說明
  - 定金支付不可退還的警告
  - 逾期支付尾款的違約處理說明
  - 購物須知確認聲明
- **FR-007**: 系統必須在訂購單頭部包含品牌標的元件（REALYOU 標誌）
- **FR-008**: 訂購單的版面排版格式必須與其他列印表單（如出貨單）保持一致
- **FR-009**: 使用者必須能夠在產生訂購單後先預覽內容，預覽以彈出視窗（ElDialog）方式呈現，與出貨單保持一致
- **FR-010**: 系統必須支援透過瀏覽器列印對話框進行列印或儲存為 PDF
- **FR-013**: 當商品明細項目較多時，系統必須使用 CSS break-inside 屬性確保每個商品項目在列印時完整不跨頁，自動分頁處理
- **FR-011**: 當訂購人的 Line ID 或銀行帳號末五碼為空時，系統必須保留對應欄位為空白
- **FR-012**: 當訂單沒有付款紀錄時，付款紀錄欄位必須顯示為 0

### Key Entities *(include if feature involves data)*

- **訂購單文件 (Order Document)**: 用於客戶確認訂單的正式文件，包含訂購人資訊、訂購明細和法律條款，以彈出視窗方式預覽，支援列印
- **訂購人資訊 (Customer Contact Info)**: 包含姓名、電話、Line ID（可選）、銀行帳號末五碼（可選），來自商品訂單資料
- **訂購明細 (Order Details)**: 根據訂單類型動態顯示，預購訂單包含品牌、商品名稱、款式、數量、單價；現貨訂單額外包含配件欄位
- **付款紀錄 (Payment Records)**: 顯示完整的付款記錄列表，每筆記錄包含付款日期、付款金額、付款方式，類似出貨單的呈現方式
- **商品預購定金須知 (Deposit Terms)**: 固定的法律條款文字，說明定金政策、退還條件、商品特性和違約處理

### Assumptions

- 訂購單的列印版面預設為 A4 紙張尺寸（210mm × 297mm）
- 付款紀錄以列表方式完整呈現，包含每筆付款的日期、金額和付款方式
- 品牌標的元件指的是 REALYOU 公司標誌（BrandBanner 元件），已存在於系統的共用元件庫中
- 訂購單的排版參考現有的 ShippingLabelPreview.vue（出貨單）格式，包含表格、標題和內容區塊的樣式
- 訂購單使用 ElDialog 元件以彈出視窗方式呈現預覽，與出貨單保持一致的使用者體驗
- 產生訂購單按鈕位於商品訂單管理頁面的工具列或操作選單中，與「產生出貨單」等按鈕並列
- **訂單資料在產生時已經過驗證，訂購單功能只負責資料呈現和列印，不處理資料驗證邏輯**
- Line ID 和銀行帳號末五碼為選填欄位，為空時在訂購單上保持空白
- 商品明細欄位根據訂單類型（預購 vs 現貨）動態調整，預購訂單不顯示配件欄位
- 列印時使用 CSS break-inside: avoid 確保商品項目完整不跨頁，由瀏覽器自動處理分頁

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 業務人員能在 5 秒內完成訂購單的產生和預覽
- **SC-002**: 訂購單必須包含所有必要欄位（訂購人姓名、電話、商品明細、付款紀錄列表、定金須知），完整度達 100%
- **SC-003**: 90% 的業務人員能在首次使用時成功產生並列印訂購單，無需額外指導
- **SC-004**: 訂購單的列印輸出必須符合 A4 紙張規格，文字和表格不超出邊界
- **SC-005**: 付款紀錄資訊的顯示準確率達 100%，每筆付款記錄的日期、金額、方式正確呈現
- **SC-006**: 訂購單的視覺格式與出貨單在排版一致性上達到 95% 相似度
- **SC-007**: 業務人員使用此功能後，訂購單製作時間相比手動編輯文件減少至少 70%
