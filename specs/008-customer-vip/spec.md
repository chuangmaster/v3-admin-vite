# Feature Specification: 客戶 VIP 會員管理

**Feature Branch**: `008-customer-vip`  
**Created**: 2026-01-28  
**Status**: Draft  
**Input**: User description: "VIP 會員管理功能，允許管理員針對特定客戶設定 VIP 會員等級、效期管理及狀態變更"

## Clarifications

### Session 2026-01-31

- Q: VIP 設定表單的 UI 互動模式（彈窗、側邊抽屜、新頁面或內嵌展開）會影響前端元件結構、路由設計和使用者體驗流暢度。 → A: 彈窗（Modal Dialog）
- Q: VIP 狀態的視覺呈現方式（徽章、純圖示、文字顏色或複合顯示）會影響 UI 設計一致性、可讀性和與 Element Plus 元件庫的整合方式。 → A: 徽章（Badge/Tag） + 文字標籤，使用 Element Plus `<el-tag>` 元件
- Q: VIP 歷程記錄的展示位置與方式（獨立區塊、摺疊面板、彈窗或時間軸）會影響元件組織、資料載入策略和頁面佈局設計。 → A: 客戶詳細頁面 - 獨立區塊（表格形式），使用 `<el-table>` 元件
- Q: 日期選擇器的使用者時區體驗（本地時區顯示自動轉換、強制 UTC 輸入、雙時區顯示或可切換時區）會影響使用者體驗、日期轉換邏輯複雜度和時區混淆風險。 → A: 使用者本地時區顯示，自動轉換，前端在送出前自動轉換為 UTC
- Q: 客戶列表頁面的 VIP 狀態計算位置（後端計算並回傳、前端計算、混合模式或快取策略）會影響 API 設計、前端效能、資料一致性和網路流量。根據 API 回應範例，後端回傳包含 `status` 欄位的 VIP 效期記錄陣列（例如：`{"status": "Active", "level": "VIP", ...}`），前端根據是否存在 `status: "Active"` 的記錄來判定客戶當前是否為 VIP 會員。 → A: 後端計算每筆記錄的 status 欄位（Active/Expired/Upcoming），前端根據是否存在 Active 記錄判定 VIP 身份

### Session 2026-01-29

- Q: 當管理員終止 VIP 會員時，規格中提到「將結束日期修改為昨日或當下」。這個決策會直接影響客戶在終止當日的 VIP 狀態和查詢邏輯。 → A: 設為當前時刻的前一秒（精確到時間戳），立即失效
- Q: 由於終止會員使用精確到秒的時間戳，我們需要明確整個 VIP 效期系統的時間精度策略。這會影響資料庫欄位設計、日期比較邏輯和查詢效能。 → A: 統一使用 TIMESTAMP，設定時預設為 00:00:00 和 23:59:59；前端送到後端時必須為 ISO String 格式
- Q: FR-005 要求檢查新效期是否與現有效期重疊，但需要明確「現有效期」的定義範圍。例如：某客戶在 2026-01-01 至 2026-12-31 是 VIP，管理員在 6 月時終止會員（end_date 改為終止時刻），之後若要設定 2026-06-15 至 2026-12-31 的新效期，是否應該被視為重疊而拒絕？ → A: 檢查所有資料庫中的紀錄（包括已過期和已終止的），完全禁止時間軸重疊
- Q: 關於 VIP 狀態的可見性和管理權限的範圍需要明確。 → A: 客戶列表和詳細頁面上的 VIP 狀態標示對所有使用者可見，無需特殊權限；僅「管理 VIP」的操作（設定會員等級、終止會員等按鈕和功能）需要 VIP 管理權限
- Q: 現在我們已經明確「查看 VIP 狀態」無需權限，但「管理 VIP」操作（設定會員等級、終止會員等按鈕）需要權限。接下來需要決定這個「VIP 管理權限」的實作粒度。 → A: 使用細粒度的分離權限控制，採用 customer.level.* 命名格式（如 customer.level.create、customer.level.terminate）
- Q: Assumptions 中提到「系統採用伺服器時區（或 UTC）作為『今日』的判定基準」，這個「或」表示尚未明確決定。時區策略會影響所有時間戳的儲存、計算和顯示邏輯，並可能影響跨時區使用場景下的正確性。 → A: 統一使用 UTC 儲存，前端依使用者時區顯示，後端計算統一用 UTC

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 設定客戶 VIP 等級與效期 (Priority: P1)

管理員需要將特定客戶升級為 VIP 會員，設定會員等級名稱和有效期間。管理員從客戶詳細資料頁面點擊「設定會員等級」按鈕，輸入等級名稱（預設為 "VIP"）、開始日期和結束日期，系統會記錄這個效期區間。

**Why this priority**: 這是 VIP 會員管理的核心功能，所有其他操作（查詢狀態、終止會員）都建立在能夠設定 VIP 等級的基礎上。沒有此功能，整個 VIP 管理模組無法運作。

**Independent Test**: 可以透過登入具有適當權限的管理員帳號，進入特定客戶的詳細資料頁面，點擊「設定會員等級」按鈕，輸入等級和效期資訊並儲存，然後驗證系統是否正確記錄並顯示 VIP 狀態來獨立測試。

**Acceptance Scenarios**:

1. **Given** 管理員在客戶詳細資料頁面且該客戶目前無任何 VIP 紀錄, **When** 點擊「設定會員等級」按鈕並輸入等級名稱 "VIP"、開始日期 2026-01-01、結束日期 2026-12-31 後儲存, **Then** 系統應建立新的 VIP 效期紀錄且客戶在此期間內顯示為 VIP 會員
2. **Given** 管理員在 VIP 設定表單中, **When** 開始日期晚於結束日期, **Then** 系統應顯示驗證錯誤訊息提示日期範圍無效
3. **Given** 管理員在 VIP 設定表單中, **When** 未填寫必填欄位（等級名稱、開始日期、結束日期），**Then** 系統應顯示驗證錯誤訊息提示必須填寫該欄位
4. **Given** 管理員設定的新 VIP 效期為 2026-06-01 至 2026-12-31, **When** 系統中已存在該客戶 2026-01-01 至 2026-09-30 的 VIP 紀錄（區間重疊，無論該紀錄是否已過期或被終止）, **Then** 系統應拒絕儲存並提示「該時段已存在 VIP 紀錄，請先調整現有效期」
5. **Given** 管理員要設定新的 VIP 效期為 2027-01-01 至 2027-12-31, **When** 系統中已存在該客戶 2026-01-01 至 2026-12-31 的 VIP 紀錄（區間不重疊）, **Then** 系統應允許建立新紀錄
6. **Given** 管理員在 VIP 設定表單中, **When** 等級名稱欄位保留預設值 "VIP" 或修改為其他文字（如 "VVIP"、"金卡會員"）, **Then** 系統應接受並儲存該等級名稱

### User Story 2 - 查詢客戶 VIP 狀態 (Priority: P1)

管理員和系統需要即時判定客戶當前是否為 VIP 會員。系統根據今日日期自動計算：若存在一筆效期區間包含今日的紀錄，則判定為 VIP 會員；否則為一般會員。

**Why this priority**: 這是 VIP 功能的核心邏輯，影響系統中所有顯示客戶 VIP 狀態的地方（客戶列表、客戶詳細頁面、服務單等）。必須在設定功能之後立即實作，確保設定的 VIP 資訊能正確顯示。

**Independent Test**: 可以透過設定不同效期的 VIP 紀錄，在不同日期檢查客戶列表和詳細頁面，驗證 VIP 狀態是否正確標示來獨立測試。

**Acceptance Scenarios**:

1. **Given** 今日為 2026-06-01，客戶有一筆 VIP 紀錄效期為 2026-01-01 至 2026-12-31（後端回傳 `status: "Active"`）, **When** 查詢該客戶資料, **Then** 系統應判定為 VIP 會員並顯示 VIP 標記
2. **Given** 今日為 2026-06-01，客戶有一筆 VIP 紀錄效期為 2023-01-01 至 2023-12-31（後端回傳 `status: "Expired"`，已過期）, **When** 查詢該客戶資料, **Then** 系統應判定為一般會員且不顯示 VIP 標記
3. **Given** 今日為 2026-06-01，客戶有一筆 VIP 紀錄效期為 2027-01-01 至 2027-12-31（後端回傳 `status: "Upcoming"`，尚未開始）, **When** 查詢該客戶資料, **Then** 系統應判定為一般會員且不顯示 VIP 標記
4. **Given** 客戶從未設定過任何 VIP 紀錄（後端回傳空陣列）, **When** 查詢該客戶資料, **Then** 系統應判定為一般會員且不顯示 VIP 標記
5. **Given** 管理員在客戶列表頁面, **When** 瀏覽客戶清單, **Then** 系統應根據每位客戶的 VIP 記錄 status 欄位自動判定 VIP 狀態並標示（若存在 `status: "Active"` 則顯示 VIP 標記）

---

### User Story 3 - 終止客戶 VIP 會籍 (Priority: P2)

管理員需要立即終止某客戶的 VIP 會員資格，例如因違規或特殊情況。管理員點擊「終止會員」按鈕，系統自動將當前有效紀錄的結束日期修改為昨日或當下，使其立即失效。

**Why this priority**: 這是重要的管理功能，但優先度低於設定和查詢。大多數情況下，VIP 會員會自然到期，只有特殊情況才需要手動終止。

**Independent Test**: 可以透過設定一個當前有效的 VIP 紀錄，點擊終止按鈕，然後驗證該客戶是否立即變為一般會員且 VIP 標示消失來獨立測試。

**Acceptance Scenarios**:

1. **Given** 今日為 2026-06-01 14:30:00，客戶有一筆當前有效的 VIP 紀錄效期為 2026-01-01 至 2026-12-31, **When** 管理員點擊「終止會員」按鈕並確認, **Then** 系統應將該紀錄的結束日期時間戳修改為 2026-06-01 14:29:59（當前時刻的前一秒），使客戶立即失去 VIP 資格
2. **Given** 管理員點擊「終止會員」按鈕, **When** 系統顯示確認對話框, **Then** 對話框應說明此操作將立即終止 VIP 資格並要求確認
3. **Given** 管理員在終止會員確認對話框中, **When** 點擊取消, **Then** 系統應關閉對話框且不執行任何操作
4. **Given** 客戶當前無任何有效的 VIP 紀錄（已過期或不存在）, **When** 管理員查看該客戶詳細頁面, **Then** 「終止會員」按鈕應不可見或禁用

---

### User Story 4 - 查看客戶 VIP 歷程記錄 (Priority: P3)

管理員需要查看客戶過去和現在的所有 VIP 會員記錄，了解該客戶的 VIP 歷程（何時成為 VIP、何時到期、是否被提前終止等）。

**Why this priority**: 這是輔助功能，對於追蹤客戶歷史和處理客訴有幫助，但不影響核心業務流程。可以在基本的設定、查詢、終止功能完成後再實作。

**Independent Test**: 可以透過為同一客戶設定多筆不同時期的 VIP 紀錄（包括已過期、當前有效、未來生效），然後在客戶詳細頁面驗證是否完整顯示所有紀錄來獨立測試。

**Acceptance Scenarios**:

1. **Given** 管理員在客戶詳細資料頁面, **When** 查看 VIP 歷程區塊, **Then** 系統應使用 `<el-table>` 元件顯示該客戶所有的 VIP 紀錄，包括等級名稱、開始日期、結束日期、狀態（Active/Expired/Upcoming）、建立時間，並按時間倒序排列
2. **Given** 客戶有多筆 VIP 紀錄（包括已過期、當前有效、未來生效）, **When** 查看 VIP 歷程, **Then** 系統應根據後端回傳的 `status` 欄位（"Active"/"Expired"/"Upcoming"）以不同樣式或標籤明確標示每筆紀錄的狀態
3. **Given** 客戶從未有任何 VIP 紀錄, **When** 查看 VIP 歷程區塊, **Then** 系統應顯示「尚無 VIP 會員記錄」的提示訊息

---

### Edge Cases

- 當客戶的 VIP 效期在今日當天到期(結束日期為今日),應如何判定?決策:若結束日期為今日,則當日仍視為 VIP 會員(FR-008 使用 `<=` 邊界包含),次日起才轉為一般會員
- 當管理員嘗試設定一個開始日期在結束日期之後的 VIP 效期時，系統應在儲存前驗證並拒絕
- 當兩個管理員同時為同一客戶設定重疊的 VIP 效期時，系統應使用資料庫層級的唯一性約束或樂觀鎖定機制防止資料不一致
- 當客戶有多筆不連續的 VIP 紀錄（例如 2025 年是 VIP，2026 年暫停，2027 年再次成為 VIP），系統應正確判定當前狀態
- 即使 VIP 紀錄已過期或已被終止，也不允許設定與該紀錄時間軸重疊的新效期。若需要重新設定該時段，必須先刪除或修改舊紀錄（此操作應受到更高權限控制或需要特殊理由）
- 當管理員終止 VIP 會員時，若同時有其他管理員正在查看該客戶資料，應如何確保即時更新？建議：前端在關鍵操作後重新載入資料，或使用 WebSocket 等即時推送機制
- 當效期跨越午夜時的邊界條件（例如結束日期為 2026-12-31 23:59:59），系統應明確定義日期比較規則（建議使用日期比較而非時間戳比較，當日 00:00:00 至 23:59:59 均視為該日）

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系統必須在客戶詳細資料頁面提供「設定會員等級」按鈕（僅對擁有 `customer.level.create` 權限的使用者顯示）
- **FR-001-1**: 系統必須在客戶列表和詳細資料頁面使用徽章（Element Plus `<el-tag>` 元件）標示 VIP 狀態，顯示皇冠圖示 + "VIP" 文字，此顯示對所有使用者可見，無需特殊權限
- **FR-002**: 系統必須以彈窗（Modal Dialog）形式提供 VIP 等級設定表單，包含以下欄位：等級名稱（文字欄位，預設值 "VIP"）、開始日期、結束日期
- **FR-002-1**: 前端日期選擇器應顯示使用者本地時區的日期，當使用者選擇日期後，前端自動將開始日期轉換為該日 00:00:00 UTC、結束日期轉換為該日 23:59:59 UTC，並以 ISO 8601 格式字串（例如：2026-01-01T00:00:00Z，其中 Z 表示 UTC）傳送至後端
- **FR-002-2**: VIP 等級設定表單的提交操作必須驗證使用者具有 `customer.level.create` 權限
- **FR-003**: 系統必須驗證 VIP 效期的日期範圍，確保開始日期不晚於結束日期
- **FR-004**: 系統必須驗證等級名稱、開始日期、結束日期為必填欄位，不可為空值；等級名稱長度不可超過 50 字元
- **FR-005**: 系統必須檢查新設定的 VIP 效期是否與該客戶資料庫中所有現有的 VIP 效期紀錄（包括已過期、已終止和當前有效的紀錄）在時間軸上重疊
- **FR-005-1**: 若新效期的時間範圍與資料庫中任一現有紀錄的時間範圍存在重疊（即兩個時間區間有任何交集），系統必須拒絕儲存並顯示錯誤訊息：「該時段已存在 VIP 紀錄，請先調整現有效期」
- **FR-005-2**: 重疊檢查邏輯為：新效期的 [start_date, end_date] 與現有紀錄的 [start_date, end_date] 若滿足 (新start_date <= 現有end_date) AND (新end_date >= 現有start_date)，則視為重疊
- **FR-006**: 系統必須持久化保存客戶的 VIP 會員資格效期資訊，包括客戶識別、等級名稱、效期起始日期、效期結束日期和紀錄建立時間
- **FR-007**: 系統必須確保同一客戶在同一時間點僅存在一筆有效的 VIP 效期紀錄
- **FR-008**: 系統必須根據當前 UTC 時間動態判定客戶的 VIP 身份：若存在一筆效期區間（start_date <= 當前 UTC 時間戳 <= end_date）包含當前時間的紀錄，則為 VIP 會員；否則為一般會員
- **FR-009**: 系統必須在客戶列表頁面自動計算並使用 `<el-tag>` 元件標示每位客戶的 VIP 狀態（顯示皇冠圖示 + "VIP" 文字）。前端根據客戶的 VIP 效期記錄中是否存在 `status: "Active"` 的記錄來判定並顯示 VIP 標示，此功能對所有使用者可見
- **FR-010**: 系統必須在客戶詳細資料頁面顯示該客戶當前的 VIP 狀態和有效期間（若為 VIP），此資訊對所有使用者可見
- **FR-011**: 系統必須提供「終止會員」按鈕（僅對擁有 `customer.level.terminate` 權限的使用者顯示，且僅當客戶當前為 VIP 會員時啟用）
- **FR-012**: 系統必須在管理員點擊「終止會員」按鈕時顯示確認對話框，說明此操作將立即終止 VIP 資格
- **FR-013**: 系統必須在管理員確認終止會員後，將當前有效紀錄的結束日期時間戳修改為當前時刻的前一秒（例如：終止時刻為 2026-06-01T14:30:00，則設為 2026-06-01T14:29:59），使其立即失效
- **FR-013-1**: 終止會員操作必須驗證使用者具有 `customer.level.terminate` 權限
- **FR-014**: 系統必須保留所有的 VIP 效期紀錄（包括已過期和被終止的），以支援歷程追蹤
- **FR-015**: 系統必須在客戶詳細頁面提供「VIP 歷程」獨立區塊，使用 `<el-table>` 元件展示客戶的所有 VIP 歷程記錄。後端 API 回應格式參考：`{"id": "uuid", "customerId": "uuid", "level": "VIP", "startDate": "ISO8601", "endDate": "ISO8601", "status": "Active|Expired|Upcoming", "createdAt": "ISO8601", "version": 1}`
- **FR-015-1**: 查看 VIP 歷程記錄需要 `customer.level.view` 權限（歷程記錄可能包含敏感的管理操作歷史）
- **FR-016**: 後端 API 必須在每筆 VIP 效期記錄中包含 `status` 欄位，其值為 "Active"（進行中）、"Expired"（已過期）或 "Upcoming"（即將生效），前端根據此欄位在表格中以不同樣式標示狀態
- **FR-017**: 系統必須記錄所有 VIP 設定、修改、終止操作的稽核日誌(由後端 API 中介層自動處理,前端無需額外實作)

### Key Entities

- **CustomerLevelPeriod (客戶會員等級效期)**: 代表客戶的會員等級資格有效期間。儲存於資料表 `customer_level_periods`。主要屬性包括：id（唯一識別碼，UUID）、customer_id（客戶 ID，外鍵，UUID）、level（等級名稱，字串，最大長度 50）、start_date（效期起始時間戳，TIMESTAMP WITH TIME ZONE 類型，儲存為 UTC）、end_date（效期結束時間戳，TIMESTAMP WITH TIME ZONE 類型，儲存為 UTC）、status（狀態，字串，值為 "Active"、"Expired" 或 "Upcoming"，由後端根據當前 UTC 時間動態計算）、created_at（建立時間，UTC）、version（版本號，整數，用於樂觀鎖定）。與 Customer（客戶）有多對一關係。前端送至後端的日期時間必須為 ISO 8601 格式字串並包含時區資訊（例如：2026-01-01T00:00:00Z，其中 Z 表示 UTC）。

- **Customer (客戶)**: 已存在的客戶實體，擴展以支援會員等級功能。透過 CustomerLevelPeriod 關聯，一個客戶可以有多筆效期紀錄（一對多關係）。

- **VipStatus (VIP 狀態)**: 計算出的狀態值，非持久化實體。表示客戶在特定時間點的 VIP 身份（VIP 會員或一般會員），根據效期紀錄的 `status` 欄位動態判定（若存在 `status: "Active"` 的記錄則為 VIP 會員）。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 管理員能在 30 秒內完成客戶 VIP 等級設定流程（包含表單填寫和儲存）
- **SC-002**: 系統能在 1 秒內計算並顯示客戶列表頁面中所有客戶的 VIP 狀態（無論資料量多少）
- **SC-003**: 系統能 100% 準確判定客戶的 VIP 身份，基於當日日期和效期紀錄進行即時計算
- **SC-004**: 系統能 100% 防止同一客戶在同一時間點存在重疊的 VIP 效期紀錄
- **SC-005**: 管理員點擊「終止會員」按鈕後，客戶的 VIP 狀態能在 2 秒內更新並反映在所有相關頁面
- **SC-006**: 系統能完整保留所有 VIP 效期歷程記錄，支援追溯任意時間點的客戶 VIP 狀態
- **SC-007**: 所有 VIP 設定、修改、終止操作都有完整的稽核日誌記錄(由後端自動處理)
- **SC-008**: 客戶列表頁面能即時標示 VIP 狀態且不影響頁面載入效能（載入時間不超過 2 秒）

## Assumptions

- 系統統一採用 UTC 作為所有時間戳的儲存和計算基準，確保跨時區一致性
- 資料庫中所有時間戳欄位（start_date, end_date, created_at）均以 UTC 儲存
- 前端日期選擇器以使用者本地時區顯示日期，使用者選擇日期後，前端自動轉換為 UTC 時間戳（開始日期設為當日 00:00:00 UTC，結束日期設為當日 23:59:59 UTC）並以 ISO 8601 格式（例如：2026-01-01T00:00:00Z）傳送至後端
- VIP 效期的時間戳範圍包含開始時間和結束時間兩端（start_date <= 當前 UTC 時間戳 <= end_date）
- 等級名稱欄位允許管理員輸入任意文字，但通常為預設的 "VIP"，或其他如 "VVIP"、"金卡會員" 等
- 同一客戶可以有多筆不連續的 VIP 效期紀錄（例如今年是 VIP，明年不是，後年又是），但任意兩筆紀錄的時間區間在時間軸上不可重疊（即使其中一筆已過期或被終止）
- 終止會員操作將結束日期時間戳精確設定為當前 UTC 時刻的前一秒，確保立即失效（例如：終止時刻為 2026-06-01T14:30:00Z，則將 end_date 設為 2026-06-01T14:29:59Z）
- VIP 狀態的判定是基於當前 UTC 時間即時計算的，不需要排程任務或背景作業來更新狀態欄位
- 前端負責在客戶列表和詳細頁面標示 VIP 狀態，後端提供判定邏輯和資料 API
- 資料庫層級的 start_date 和 end_date 欄位必須使用 TIMESTAMP WITH TIME ZONE 類型以支援精確到秒的 UTC 時間戳比較，確保終止會員操作能立即生效且跨時區一致
- 後端接收前端傳送的 ISO 8601 格式字串後，應解析為 UTC 時間戳儲存至資料庫；若前端傳送的時間戳包含非 UTC 時區資訊，後端應先轉換為 UTC 再儲存
