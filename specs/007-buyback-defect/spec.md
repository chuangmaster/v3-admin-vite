# Feature Specification: 收購單瑕疵欄位

**Feature Branch**: `007-buyback-defect`  
**Created**: 2026-01-29  
**Status**: Draft  
**Input**: User description: "仿效寄賣單，收購單也需要有瑕疵欄位的填寫，也需要送到後端去，規格完全仿效寄賣單"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 收購單商品項目填寫瑕疵資訊 (Priority: P1)

店員在建立或修改收購單時，需要針對每件收購商品記錄商品的瑕疵狀況，以確保交易透明度並留存商品狀況記錄供後續查詢與爭議處理。

**Why this priority**: 瑕疵記錄是商品狀況的重要證明，直接影響交易價格評估、客戶糾紛處理與法律證據保全，為核心業務需求。

**Independent Test**: 可透過建立一筆收購單，在商品項目中選擇多個瑕疵項目（如五金刮痕、皮質磨損），儲存後驗證資料正確提交至後端，並在查詢介面確認瑕疵資訊正確顯示來獨立測試。

**Acceptance Scenarios**:

1. **Given** 店員正在填寫收購單的商品項目表單，**When** 點擊商品瑕疵處欄位，**Then** 系統顯示多選選項：五金生鏽/刮痕/掉、皮質磨損/刮痕/壓痕、內裡髒污、四角磨損
2. **Given** 店員查看瑕疵選項清單，**When** 選擇一個或多個瑕疵項目（如勾選「五金生鏽/刮痕/掉」和「皮質磨損/刮痕/壓痕」），**Then** 系統記錄選擇的瑕疵項目並在表單中顯示已選擇的項目
3. **Given** 店員已選擇部分瑕疵項目，**When** 取消勾選某個已選擇的瑕疵項目，**Then** 系統移除該瑕疵項目並更新表單顯示
4. **Given** 店員的商品確實無瑕疵，**When** 不勾選任何瑕疵項目並提交表單，**Then** 系統允許提交（瑕疵欄位為選填，空白代表無瑕疵）
5. **Given** 店員完成收購單表單填寫（包含瑕疵資訊），**When** 提交表單至後端，**Then** 系統將每個商品項目的瑕疵資訊（以陣列形式）與其他商品資料一起提交至後端 API
6. **Given** 店員儲存收購單後查詢該筆記錄，**When** 進入收購單詳細頁面，**Then** 系統在每個商品項目中正確顯示已記錄的瑕疵項目清單

---

### User Story 2 - 查詢與匯出包含瑕疵資訊 (Priority: P2)

店員在查詢收購單時，需要看到商品的瑕疵記錄，並在匯出 Excel 時包含瑕疵資訊，以便進行統計分析與報表製作。

**Why this priority**: 瑕疵資訊的查詢與匯出是輔助功能，支援數據分析與管理報表，但不影響核心建單流程（P1 已包含瑕疵資訊的記錄與顯示）。

**Independent Test**: 可透過查詢包含瑕疵資訊的收購單，確認詳細頁面正確顯示瑕疵清單，並匯出 Excel 檔案驗證瑕疵欄位正確呈現來獨立測試。

**Acceptance Scenarios**:

1. **Given** 店員在收購單查詢結果列表中，**When** 點擊某筆收購單查看詳細資訊，**Then** 系統在商品項目區域顯示該商品的瑕疵清單（如「五金生鏽/刮痕/掉、皮質磨損/刮痕/壓痕」）
2. **Given** 店員需要匯出收購單報表，**When** 點擊「匯出 Excel」按鈕，**Then** 系統生成 Excel 檔案，在商品項目列中新增「商品瑕疵處」欄位，內容為瑕疵項目清單（以頓號「、」分隔，如「五金生鏽/刮痕/掉、皮質磨損/刮痕/壓痕」）
3. **Given** 某筆收購單的商品項目無瑕疵記錄，**When** 匯出該筆收購單至 Excel，**Then** 該商品項目的「商品瑕疵處」欄位顯示為空白或「無」

---

### User Story 3 - 修改收購單的瑕疵資訊 (Priority: P3)

店員在發現瑕疵記錄錯誤或遺漏時，需要能夠修改已儲存收購單的瑕疵資訊，確保記錄準確性。

**Why this priority**: 修改功能提升資料準確性，但非必要核心流程，可在初始版本後優化。

**Independent Test**: 可透過進入已儲存的收購單詳細頁面，修改商品項目的瑕疵選項（新增或刪除瑕疵項目），儲存後驗證修改記錄並確認後端資料更新來獨立測試。

**Acceptance Scenarios**:

1. **Given** 店員在收購單詳細頁面並有修改權限，**When** 點擊編輯按鈕進入編輯模式，**Then** 系統允許修改商品項目的瑕疵欄位（增加或移除瑕疵項目）
2. **Given** 店員修改瑕疵資訊並儲存，**When** 提交修改，**Then** 系統更新後端資料並記錄修改歷史（修改者、修改時間、修改內容）

---

### Edge Cases

- 若商品項目的瑕疵選項在未來需要新增或修改（如新增「拉鍊損壞」選項），系統應支援選項配置更新而不影響已儲存的歷史記錄（已儲存的瑕疵項目應保留原始名稱）
- 若後端 API 回傳的瑕疵資料格式不符預期（如空值、非陣列格式），前端應進行容錯處理並顯示「資料格式錯誤」或顯示為空白
- 若店員在修改收購單時刪除所有瑕疵項目，系統應將瑕疵欄位儲存為空陣列，代表該商品無瑕疵
- 若匯出 Excel 時單一商品項目選擇了所有瑕疵項目，瑕疵欄位內容應完整顯示所有項目名稱（以頓號分隔），不應截斷或省略
- 若瑕疵項目名稱中包含特殊字元（如斜線「/」），Excel 匯出時應正確處理，避免格式錯誤

## Requirements *(mandatory)*

### Functional Requirements

#### 瑕疵欄位填寫與驗證

- **FR-001**: 系統必須在收購單的每個商品項目表單中提供「商品瑕疵處」多選欄位，選項包含：五金生鏽/刮痕/掉、皮質磨損/刮痕/壓痕、內裡髒污、四角磨損
- **FR-002**: 系統必須允許店員針對每個商品項目選擇零個或多個瑕疵項目（瑕疵欄位為選填，空白代表無瑕疵）
- **FR-003**: 系統必須在店員勾選或取消勾選瑕疵項目時，即時更新表單顯示的已選擇項目清單
- **FR-004**: 系統必須在收購單表單提交時，將每個商品項目的瑕疵資訊以陣列格式（如 `["五金生鏽/刮痕/掉", "皮質磨損/刮痕/壓痕"]`）一併提交至後端 API
- **FR-005**: 系統必須在後端 API 接收瑕疵資訊後，將其與商品項目資料關聯儲存，確保每個商品項目的瑕疵記錄獨立且完整

#### 瑕疵資訊查詢與顯示

- **FR-006**: 系統必須在收購單詳細頁面的商品項目區域顯示該商品的瑕疵清單，若無瑕疵則顯示「無」或留空
- **FR-007**: 系統必須在瑕疵清單顯示時，以易讀的格式呈現（如以頓號「、」分隔的文字清單：「五金生鏽/刮痕/掉、皮質磨損/刮痕/壓痕」）

#### Excel 匯出

- **FR-008**: 系統必須在收購單匯出 Excel 功能中，針對每個商品項目新增「商品瑕疵處」欄位
- **FR-009**: 系統必須在「商品瑕疵處」欄位中，將該商品的瑕疵項目以頓號「、」分隔的文字格式顯示（如「五金生鏽/刮痕/掉、皮質磨損/刮痕/壓痕」），若無瑕疵則顯示空白或「無」
- **FR-010**: 系統必須確保 Excel 匯出時，瑕疵資訊與其他商品項目欄位（品牌名稱、款式、內碼、金額等）在同一列正確對齊

#### 瑕疵資訊修改

- **FR-011**: 系統必須允許具有修改權限的店員進入收購單編輯模式，修改商品項目的瑕疵資訊（新增或移除瑕疵項目）
- **FR-012**: 系統必須在店員修改瑕疵資訊並儲存時，更新後端資料並記錄修改歷史（包含修改者、修改時間、修改欄位與修改前後值）

#### 資料容錯與驗證

- **FR-013**: 系統必須在前端驗證瑕疵資料格式（確保為陣列且項目為有效的瑕疵選項），若格式不符則顯示錯誤訊息並阻止提交
- **FR-014**: 系統必須在後端 API 驗證瑕疵資料格式與內容，拒絕無效或惡意的資料（如非預期的選項值、超長字串等），並回傳明確的錯誤訊息
- **FR-015**: 系統必須在查詢或顯示瑕疵資訊時，對後端回傳的異常資料（如空值、非陣列格式）進行容錯處理，避免前端崩潰，顯示「資料格式錯誤」或留空

### Key Entities

- **商品項目（Product Item）**: 在收購單中新增「商品瑕疵處」屬性，儲存格式為字串陣列（如 `["五金生鏽/刮痕/掉", "皮質磨損/刮痕/壓痕"]`），用於記錄該商品的瑕疵狀況清單。空陣列代表無瑕疵。與收購單為多對一關係（一筆收購單可有 1-4 個商品項目）。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 店員能在建立收購單時，在 30 秒內完成所有商品項目的瑕疵資訊選擇（每個商品項目平均 10 秒）
- **SC-002**: 瑕疵資訊的前端到後端提交成功率達到 99% 以上（排除網路異常情況）
- **SC-003**: 收購單查詢詳細頁面能在 1 秒內正確顯示商品項目的瑕疵資訊（包含多個瑕疵項目的情況）
- **SC-004**: Excel 匯出功能能正確包含瑕疵欄位，且瑕疵資訊格式易讀、無格式錯誤（如欄位對齊錯誤、特殊字元顯示異常等）
- **SC-005**: 90% 的店員能在首次使用時，無需額外指導即能找到並正確使用瑕疵欄位功能
- **SC-006**: 瑕疵資訊修改功能的資料更新成功率達到 99% 以上，且修改歷史記錄完整無遺漏
